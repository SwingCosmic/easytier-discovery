# 待讨论分歧点

本文档只记录仍有较大变数、需要多轮讨论后再冻结的设计项。

## 1. 语言与实现形态

当前倾向：

- 核心逻辑使用 Rust。
- 控制面服务使用独立进程。
- 多语言接入在首版优先 Node.js、Java、.NET。

待确认问题：

- Rust 是否只作为核心算法库，还是连 A 类注册中心也统一用 Rust 实现。
- 首版是否要同步产出标准化协议定义，例如 protobuf/IDL。
- 是否需要在首版就保留完整插件扩展点，例如自定义评分器、自定义健康检查器。

## 2. sidecar 进程模式 vs C ABI 包装 SDK

### 2.1 sidecar/agent 模式

优点：

- 语言无关，适配微服务快。
- 进程隔离更好，崩溃影响更小。
- 升级与回滚可独立于业务应用。
- 更容易统一运维、日志、观测与权限控制。

缺点：

- 本地 IPC 多一跳。
- 需要管理生命周期、守护、重连和版本兼容。
- 容器、桌面和移动端的部署手法不完全一致。

更适合：

- Node.js
- Java
- 已有微服务或单体应用快速接入

### 2.2 C ABI/FFI 模式

优点：

- 调用路径短。
- 容易共享进程内状态与业务指标。
- 对性能敏感服务更直接。

缺点：

- 跨语言 ABI 设计复杂。
- 崩溃隔离更差。
- 升级与发布节奏被业务应用绑定。
- Node.js、Java 等语言的构建、分发和平台兼容成本更高。

更适合：

- Rust
- C/C++
- .NET PInvoke
- 少数对时延极敏感的场景

### 2.3 当前建议

首版建议不是二选一，而是明确主次：

- Node.js、Java：主推 sidecar
- .NET：可双轨评估，先把 sidecar 作为保底
- Rust/C/C++：可优先直连核心库

需要进一步拍板的问题：

- sidecar 的控制协议是否只保留 gRPC，还是同时提供 HTTP/JSON 便于调试
- sidecar 是否负责启动 EasyTier runtime，还是只连接外部已存在实例
- sidecar 与业务进程的权限边界如何划分

## 3. A 类注册中心的职责上限

待讨论：

- A 是否只做目录聚合与评分，还是允许承担代理转发
- A 是否应该内建配置存储，还是先把持久化抽象成接口
- A 是否要承担更多区域协调职责，例如冲突合并建议、策略分发和审计

当前倾向：

- 首版允许 A 做兜底中转，但不要把它做成强耦合网关
- 先保留持久化接口，不急着绑定单一数据库

## 4. 评分与策略的可配置程度

待讨论：

- 权重矩阵是否固定内置，还是按命名空间/服务可配置
- `latency_first`、`stability_first` 等策略是配置枚举，还是允许脚本化扩展
- 是否需要把网络策略和业务策略拆成两层，以免业务方误改核心可用性逻辑

当前倾向：

- 首版先内置少量稳定策略
- 权重允许有限配置，但不开放任意脚本扩展

## 5. Actor / Serverless 的首版范围

待讨论：

- 首版是否只预留数据模型，不做真正的 placement
- Actor placement 是控制面能力，还是独立子系统
- Actor sticky、迁移和 activation lease 是否会把当前设计复杂度拉高过多

当前倾向：

- 先把 Actor 当成应用层扩展方向，不放进首版闭环核心

## 6. 移动端支持的深度

待讨论：

- 首版是否只做字段预留
- 后续移动端是嵌入式 SDK 形态，还是“移动端只做 C 角色调用方”
- iOS/Android 上 EasyTier 生命周期和 TUN 权限由谁管理

当前倾向：

- 首版不做正式移动端支持，只保留模型边界和打包预研事项
